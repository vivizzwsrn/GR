PART 01

建设背景与核心理念

从分散支撑走向一体化运营管控

⚓ 运营升级方向

随着港口物流业务持续复杂化，合同、订单、计费、支付、客服反馈等数据仍分散在不同系统中，状态链不完整、口径不统一、跨系统追溯困难，现有人工核对和事后补救方式已难以支撑精细化运营。

🔴 被动响应模式
→→→
🟢 主动管控模式

本方案采用“履约中心业务应用 + 履约数据支撑 + 运营管理数据中心”协同建设模式：先保障履约业务闭环与上线取数，再统一开展跨系统数据治理、资产目录和场景服务。

📊

数据贯通 · 运营可视

拉通合同、订单、箱号、费用、支付、结算和履约节点，让每一票业务可穿透查询，支撑运营监控与经营分析。

🛡️

风险前置 · 主动预警

基于状态一致性、金额一致性、履约超时、规则冲突等校验机制，将异常识别从事后追查前移到过程预警。

🤝

流程闭环 · 协同处置

通过工单闭环打通商务、业务、客服、财务和管理端，让异常发现、派单、处理、复核、归档形成统一流程。

🏗️

底座夯实 · 标准治理

按大数据项目标准动作建设数据标准、主数据、质量规则、每日快照、血缘追溯和安全权限，避免只做上层展示。

PART 03

履约中心定位与建设内容

履约应用负责流程闭环 · 运营管理数据中心负责企业级治理

业务执行引擎

围绕“协议合同－订单－执行－计费结算－异常反馈－归档”建立流程闭环

履约中心定位为业务应用和流程闭环系统，负责履约业务域的在线化管理与过程控制，不单独承担企业级数据治理职责。其运行过程中沉淀的合同、计费、履约、结算、工单和操作日志，将作为运营管理数据中心的重要数据源。

协议合同 订单执行 履约节点 计价计费 工单归档
👉 边界原则：履约数据支撑层只解决业务运行取数和状态沉淀；运营管理数据中心统一承接企业级主数据、元数据、质量、安全、资产目录和数据服务。
01

合同管理

统一管理合同模板、签章状态、变更记录、附件归档和关键日期提醒，打通合同与订单、费用、履约的关联。

02

计价计费

承接费率配置、计算单、调整单、应收应付、补缴退款和发票校验，减少漏收、错收和人工复核。

03

履约进度

按合同号、订单号、箱号、计划号跟踪关键节点，形成履约时间轴、SLA预警和业务过程留痕。

04

对账结算与异常闭环

贯通对账单、核销、发票和异常反馈；规则命中后形成待核查清单或工单，跟踪处理、复核与归档。

客户服务端

提供合同查询、履约进度、费用明细和发票查询，减少人工查询和业务进度沟通成本。

内部管控端

面向业务、商务、客服、财务和管理人员提供合同、费率、履约计划、计费、结算、异常和工单等内部管理能力。

01

履约中心应用

客户服务端与内部管控端共同承接业务操作、流程审批和异常处置。

02

履约数据支撑层

沉淀上线必需的合同、订单、费用、结算、节点、异常、工单和日志。

03

运营管理数据中心

统一接入履约及相关系统数据，完成治理、资产化、服务发布和场景分析。

PART 07

数据治理全流程：把原始数据生产为可信资产

不是“随便抽取、简单整合”，而是标准先行、分层加工、质量闸门、全程留痕的受控数据生产体系

◆ 核心判断

数据接入只是起点；只有经过标准、模型、质量、安全和运营控制，数据才能进入经营分析与业务应用。

港口数据来自合同、订单、箱货、计费、支付、履约、工单等多个系统。治理的关键不是把表“搬到一起”，而是建立统一业务语义、稳定关联主键、可审计加工链路和可量化质量规则，确保每一项指标都能追溯到来源、口径和责任人。

标准定义＋分层建模＋质量闸门＋安全权限＋持续运营

一份数据可进入经营应用，至少满足五项条件

  • ✓定义一致：指标名称、业务口径、统计粒度和时间窗口已确认。
  • ✓来源可溯：能定位到源系统、源表、字段及采集批次。
  • ✓过程可审：清洗、转换、关联、汇总逻辑有版本和发布记录。
  • ✓质量可量：完整性、准确性、一致性、唯一性、有效性、及时性可校验。
  • ✓责任可找：数据 Owner、开发责任人和问题处理时限明确。
🧭

治理规划与盘点

按业务域盘点系统、表、接口、指标和敏感数据，明确治理优先级及责任边界。

关键控制：数据域 / 主题域、Owner、系统权威源、共享边界、SLA。
主要产物：资产清单、治理台账、源系统矩阵。
📐

标准与口径设计

统一订单、客户、箱号、费用、支付、履约节点等核心对象的名称、编码和统计口径。

关键控制：字段标准、码值、单位、时间格式、主数据、指标公式。
主要产物：数据标准、指标字典、码表、映射规则。
🔌

受控接入与落地

通过数据源与同步任务接入全量 / 增量数据，原样落入 ODS 层并保留采集批次。

关键控制：网络连通、字段映射、水位线、分区、对账、Schema 变更。
主要产物：接入任务、ODS 原始表、采集日志。
🧹

清洗与标准化

处理空值、重复、错误码、异常格式和单位差异，并保留问题数据及修正规则。

关键控制：去重主键、空值策略、类型转换、码值映射、异常隔离。
主要产物：标准明细、拒绝记录、清洗规则库。
🧩

主数据对齐与融合

以统一客户、订单、合同、箱号等主键建立跨系统关联，解决一物多码和主体不一致。

关键控制：黄金记录、主键映射、生效时间、冲突优先级、匹配置信度。
主要产物：主数据、统一维表、关联关系表。
🏗️

分层建模与加工

按 ODS → DIM / DWD → DWS → ADS 分层加工，明细、汇总和应用口径相互隔离。

关键控制：事实粒度、维度一致性、拉链表、分区、生命周期、依赖关系。
主要产物：事实表、维表、主题宽表、指标结果表。
🛡️

质量闸门与安全控制

规则随调度触发；强规则失败时阻断下游，敏感字段按分类分级授权和脱敏。

关键控制：阈值、强弱规则、告警、权限审批、脱敏、下载审计。
主要产物：质量报告、异常工单、权限与审计记录。
♻️

发布服务与持续运营

通过目录、报表或 API 发布，监控时效、成本和使用情况，并用问题闭环持续优化规则。

关键控制：血缘影响分析、版本发布、基线、重跑、下线、使用评价。
主要产物：数据目录、服务 API、SLA 报告、改进清单。

质量闸门与责任机制

以六类规则守住数据入口，并将“谁定义、谁实现、谁审批、谁处置”落到责任人。

完整性缺失即告警
准确性异常先隔离
一致性冲突有仲裁
唯一性重复则阻断
有效性非法值拦截
及时性延迟触发预警
关键规则失败：阻断下游 · 一般异常：告警并派单 · 处置过程：全程留痕
业务数据 Owner定义口径与最终裁决
数据管理员维护标准与治理台账
开发与运维实现规则、监控与恢复
安全与应用责任人审批权限与数据使用

数据集成（Data Integration）

把数据稳定、可核对地搬进数仓
接入

负责库表、API 和文件等异构数据源的全量或增量同步；所有接入任务均在统一项目空间内配置资源、网络、失败告警和重跑策略。

港融项目用法：创建 TOS / 订单 / 财务等数据源 → 选择源表与 MaxCompute ODS 目标表 → 配置字段映射、分区 ${bizdate}、并发数和脏数据策略 → 入仓即执行首轮质量校验。
🧮

MaxCompute

海量数据的存储与批量计算底座
存算

作为离线数仓和批量计算底座，负责数据表、分区、SQL 计算、生命周期与每日快照，并由 DataWorks 统一开发和调度。

港融项目用法：ODS 保留源数据 → DIM / DWD 统一客户、订单、箱货和费用明细 → DWS 汇总主题指标 → ADS 输出看板、核验和异常识别结果。
🧱

数据开发（DataStudio）

开发 SQL、依赖和周期任务
开发

在统一工作空间中开发 MaxCompute SQL 和工作流，配置上下游 DAG、调度参数、质量规则与运行资源，并完成测试、评审和生产发布。

港融项目用法：按业务流程建立工作流；使用 ${bizdate} 接收业务日期；以“接入 → 清洗 → 维表 / 明细 → 汇总 → 应用”连线，标准模式下经开发、评审后发布生产。
🛡️

数据质量与生产运维

Data Quality + Operation Center
保障

质量规则覆盖分层表和服务结果，随调度节点执行；运维中心通过 DAG、实例、基线和日志监控任务，支持告警、诊断、补数和重跑。

港融项目用法：关键订单、支付和计费强规则失败时阻断下游，一般问题告警并派单；为“每日经营数据产出”配置基线并输出达标率。
🗺️

数据地图 + 数据服务

让数据找得到、看得懂、取得到、用得稳
服务

数据地图统一沉淀数据源、Owner、元数据、字段口径、血缘、质量和安全等级；数据服务将治理达标且获得授权的结果表发布为标准 API。

港融项目用法：业务人员先在目录检索指标与血缘并申请权限；成熟的订单状态、箱货轨迹、费用核验和风险提示结果，通过审批后向履约中心或授权应用提供服务。
📊

Quick BI / DataV

经营分析、报表门户与可视化大屏
展现

Quick BI 面向数据集、自助分析和仪表板；DataV 侧重可视化大屏与交互编排。二者只消费治理达标数据，并服从字段权限、脱敏、导出和审计要求。

港融项目用法：Quick BI 承载领导经营看板、业财分析、客户分析和专题报表；DataV 承载港口运营驾驶舱、船箱轨迹、场站态势与风险预警大屏。展示层只读取通过质量校验和授权的数据集。
按需扩展边界：近实时箱动态或日志类需求，可在数据规模、时效和并发要求明确后扩展实时或查询加速链路；新增能力仍须纳入同一元数据、质量、调度与权限体系，基础组合不预设额外查询加速组件。产品范围需结合采购版本和总体成本确认。

异常类型 规则口径 输出结果
字段质量异常 客户主体、合同号、订单金额、支付时间、箱号 / 计划号、业务来源、签章状态、客服工单号等关键字段为空、重复或格式不规范。 字段缺失清单、重复记录、来源系统、发现批次
支付状态异常 支付成功但缺少流水，支付失败但业务已放行，退款 / 补缴状态与订单状态不一致。 状态冲突清单、异常标签、规则编号
履约节点异常 关键节点超时、流程跳转不符合规则、合同状态与订单状态不一致、履约结果未回传。 履约预警清单、处置建议、节点快照
金额与流水异常 订单应收、实收、退款、补缴、减免、发票金额与流水、渠道记录不一致。 金额差异清单、差异金额、来源和核查口径
规则治理要求：异常阈值、强弱规则、启停时间和责任人需经业务确认；规则变化保留版本记录，避免口径漂移。
处置闭环要求：系统提供工单建议但不替代业务判断；处理人、处理时间、原因分类和复核结果需要结构化留痕。

PART 12 · 长期演进

实施路径｜五年演进规划

业务数据汇集 → 全域融合 → AI赋能 → 服务开放 → 商业变现

第1年 · 业务数据汇集 + 链路透视
全链路查看与异常发现
优先承接集装箱等核心业务数据。实现从业务订单到财务结算的数据全链路透明化，重点构建异常数据（如状态停滞、未执行退款等）的自动化发现机制，迈出数字化内管的第一步。
第2年 · 全域数据融合 + 量化考核
全域接入与管理量化
陆续接入港区各业务系统数据，构建统一数据资产池。围绕员工任务处理、异常闭环、处理时效等真实业务指标，建立量化分析与考核机制，推动内部管理由经验驱动向数据驱动转变；同时基于客户交易、业务频次、价值贡献等数据开展客户 RFM 模型分析，提升内部管理精细化水平和客户经营能力。
第3年 · AI赋能 + 高质量数据集
精细化运营能力构建
基于统一数据底座，完善异常识别、任务分派、责任追踪、处理留痕和绩效分析能力。重点解决人员执行效率、业务协同和问题闭环管理问题，形成可统计、可考核、可优化的内部运营管理体系；同时围绕业务趋势预测、异常风险识别、经营分析等场景，引入 AI 辅助分析能力，形成可复用的数据分析模型和高质量数据集，为管理决策、业务优化和智能应用提供支撑。
第4年 · 服务开放 + 数据赋能
数据服务标准化与 API 输出
依托扎实的内控基座，将成熟的业务查询、全程时效追踪、车辆核验等能力封装为标准API，正式向船公司、货代、车队及口岸单位输出开放，实现从内部管理支撑向外部生态赋能转变。
第5年 · 模式升级 + 商业变现
数据变现与生态运营
深度探索数据要素的市场化价值。通过企业高级数据订阅、产融风控协同（如信用数据对接供应链金融）等手段，实现数据资产的直接商业变现，并跨港区复制“港融模式”。
👉 规划边界：第 1 年以 M1—M4 实施与验收基线为准；第 2—5 年属于战略演进方向，年度范围需结合业务优先级、数据源条件、资源投入和阶段评估滚动确认。
